# Using ILA with vivado-bridge

A practical guide for adding ILA (Integrated Logic Analyzer) cores
to a design, arming triggers, and capturing waveforms through this
bridge -- entirely from Python/Tcl, no Vivado GUI required.

> Scope: capture-and-analyze workflows. The Vivado IDE waveform
> viewer is GUI-only (same constraint as the VIO Dashboard) and is
> deliberately not part of this flow. Captured data goes to CSV/VCD
> for the AI / Python side to analyze.

> **Quick path**: the runtime side of this flow (configure / set
> trigger / arm / wait / export CSV / parse) is now covered by the
> `ila.*` dispatcher ops. Read [op_ila.md](op_ila.md) for the JSON
> API; this document is the design-time companion (mark_debug, debug
> XDC, trigger syntax, capture buffer behaviour, the ILA pitfalls
> below). When you're at "I have a programmed device with an ILA
> visible and I want a CSV", reach for the `ila.*` ops via
> `vivado_op.py` rather than hand-writing
> `set_property TRIGGER_COMPARE_VALUE ...` snippets.

## Why a separate document

`operations/debug.py` covers VIO and ILA enumeration; `operations/ila.py`
covers the runtime capture flow. The *design-time* side of ILA --
how the core gets *into* the design at all -- still has Vivado/Tcl
quirks that no Python wrapper can sand off, and they need a guide:

- `mark_debug` Verilog attributes alone do not insert an ILA -- they
  only mark nets so a *separate* `create_debug_core` step can pick
  them up.
- The XDC autogenerated by `save_constraints` after
  `create_debug_core` is missing the ILA's clock connection (a
  consistent Vivado 2024.1 issue we hit in real bring-up).
- Trigger compare values use a non-Verilog edge syntax (`eq1'bR`
  for "rising edge") and need `{...}` Tcl bracing to survive shell
  quoting.
- Naming collides with VIO when the same wire feeds both -- VIO probe
  names get renamed (e.g. `mode_select` becomes `mode_select_2`).
- `wait_on_hw_ila` can hang far longer than its `-timeout` value
  suggests; polling `STATUS.CORE_STATUS` is more reliable.

This document assumes you've read [using_vio.md](using_vio.md) -- the
VIO + ILA combined workflow is the most useful one for AI-driven
testing, and it relies on knowing how VIO behaves first.

## Why ILA is a great fit for AI-driven debugging

Unlike VIO, ILA is **inherently a "capture data, analyze later"**
tool. Vivado's official Tcl flow is built around exporting captured
data to CSV/VCD, not around GUI inspection. That makes it a first-
class citizen for headless workflows:

- AI sets up trigger conditions in Tcl
- AI arms the ILA, drives the design via VIO, then captures
- AI exports CSV, parses with `csv` / `pandas`, asserts properties
  ("rising edge of signal X happened within 4 cycles of mode change",
  "counter increments monotonically", "FSM never enters the X state")

The GUI VIO/ILA dashboards are convenient *for humans*, but the AI
loop doesn't need them. The capture window of a typical 4096-sample
ILA at 125 MHz is 32.8 us -- short enough that anything observable
with the human eye (1 Hz LED blink, button bounce, second-scale
stuff) won't fit. ILA is for **per-clock-cycle precision** debug.

## 1. Choosing what to observe

ILA observes signals at the design clock rate. At 125 MHz with
4096-sample depth that's a 32.8 us window. **Slow signals don't fit.**

| Signal class | Period | Fits in 4096-sample window? |
|---|---|---|
| `clk_div_1Hz_toggle` | 1 s | No (would need 125e6 samples) |
| 1 kHz PWM | 1 ms | Marginal (window covers ~3% of one period) |
| UART byte at 115200 | 87 us | Almost (1 byte ~ 10880 cycles) |
| Per-clock counter LSB | 8 ns | Yes (lots of cycles) |
| State machine ticking each clock | 8 ns | Yes |
| VIO write -> fabric response | A few clocks | Yes (and ILA shines here) |

For initial bring-up of an unfamiliar design, the most useful pattern
is: **drive a control signal via VIO, trigger ILA on its edge,
capture the surrounding window.** That gives you the full
input/output relationship at clock-cycle resolution.

## 2. Marking signals for debug

Add `(* mark_debug = "true" *)` to the `wire` / `reg` declarations
you want to observe. The attribute is a *hint* -- it tells Vivado
to keep the net through optimization and expose it as a debug
candidate. It does **not** create an ILA on its own.

```verilog
(* mark_debug = "true" *) reg  [7:0] fast_counter;
(* mark_debug = "true" *) reg  [1:0] state_fast;
(* mark_debug = "true" *) wire       mode_select;
(* mark_debug = "true" *) wire [3:0] led_internal;
```

Verify after synthesis via the `exec_tcl.py` escape hatch:

```bash
python exec_tcl.py "open_run synth_1 -name synth_1"
python exec_tcl.py "get_nets -hier -filter {MARK_DEBUG==1}"
# Response output:
#   fast_counter[0] fast_counter[1] ... mode_select led_internal[0] ...
```

If a signal you marked doesn't appear, it was probably optimized
into a constant or merged with another net during synthesis.

### Pre-trap: a register with no fanout is removed entirely

Before reaching the "ILA didn't insert" trap below, there's a step-0
trap: if a register's only purpose is to be observed (the canonical
example is a 16-bit `count` driven internally with no port connection),
**Vivado's synthesis removes the register completely** without
`mark_debug`. The signs you've hit this:

```bash
python exec_tcl.py "get_cells -hier <reg_name>"    # empty -> register removed
python exec_tcl.py "get_nets -hier <reg_name>*"    # empty -> nets gone too
```

Adding `(* mark_debug = "true" *)` resurrects the register: after
re-synth, `get_cells -hier count_reg*` returns the FFs and
`get_nets -hier count*` returns the bit nets `count[0]..count[N-1]`
that you can then connect to ILA probes. This is observation #1
when bringing up an ILA on a brand-new design where the signal you
want to debug has no external visibility.

> **You are not done yet.** `mark_debug` is necessary but not
> sufficient -- the next section adds the actual ILA core. Skipping
> §3 and going straight to `build.implement` produces a bitstream with
> `ila_count=0`. (See §10b for why this trap is so easy to fall into.)

## 3. Inserting an ILA core

> **Critical**: `mark_debug` alone (the attribute from §2) does **not**
> insert an ILA on Vivado 2024.1 -- it only keeps the net through
> synth. You also need an explicit `create_debug_core` step (this
> section). See §10b for the full footgun details and the recommended
> dedicated-XDC pattern.

Two-step Tcl flow on the open synthesized design:

```tcl
# 1. Create the core and configure
create_debug_core u_ila_0 ila
set_property C_DATA_DEPTH 4096       [get_debug_cores u_ila_0]
set_property C_TRIGIN_EN  false      [get_debug_cores u_ila_0]
set_property C_TRIGOUT_EN false      [get_debug_cores u_ila_0]

# 2. Connect the clock
set_property port_width 1 [get_debug_ports u_ila_0/clk]
connect_debug_port u_ila_0/clk [get_nets clk]

# 3. Connect each probe (probe0 is auto-created; use create_debug_port for more)
set_property port_width 1 [get_debug_ports u_ila_0/probe0]
connect_debug_port u_ila_0/probe0 [get_nets mode_select]

create_debug_port u_ila_0 probe
set_property port_width 4 [get_debug_ports u_ila_0/probe1]
connect_debug_port u_ila_0/probe1 \
    [get_nets [list led_force[0] led_force[1] led_force[2] led_force[3]]]

# ... repeat for additional probes ...

# 4. dbg_hub clock frequency -- override the 300 MHz default
set_property C_CLK_INPUT_FREQ_HZ 125000000 [get_debug_cores dbg_hub]
set_property C_USER_SCAN_CHAIN 1           [get_debug_cores dbg_hub]
connect_debug_port dbg_hub/clk             [get_nets clk_IBUF_BUFG]
```

Notes:
- `port_width` must match the bus width *exactly*. A 4-bit bus on a
  1-bit probe fails at impl time with `[Chipscope 16-213]`.
- **`dbg_hub` defaults to `C_CLK_INPUT_FREQ_HZ = 300000000` (300 MHz).**
  On a 125 MHz design this is wrong: Vivado will tolerate it but
  produces a timing-skew warning, and the constraints generated for
  the debug hub clock domain will be off. Always override to the
  actual clock frequency, as in step 4 above. (The exact net name
  -- `clk_IBUF_BUFG` here -- depends on your design; check the post-
  synth netlist with `get_nets -of_objects [get_pins <inst>/clk]`.)
- After all `connect_debug_port` calls, **save the constraints** so
  the debug core survives a synth_1 reset:
  ```tcl
  save_constraints -force
  ```
  Vivado writes the debug core definition into a fresh XDC file (e.g.
  `<project>.srcs/constrs_1/new/<top>.xdc`).

> **Watch out** (raw-Tcl path only): `save_constraints -force` writes
> into the XDC currently selected by `target_constrs_file` -- which by
> default is the **first** XDC in `constrs_1`, typically the
> hand-authored `pynq_z1.xdc`. Calling `save_constraints` after
> `create_debug_core` therefore appends the debug core lines into the
> file you are maintaining by hand, and a later `delete_debug_core`
> leaves dbg_hub residue behind.
>
> `debug.create_ila_core` (see §10b) routes the save into a dedicated
> `debug_<name>.xdc` automatically and restores `target_constrs_file`
> on the way out, so this footgun applies only when you skip the
> wrapper and drive raw Tcl.

### Watch out: missing `u_ila_0/clk` line in the autogenerated XDC

In Vivado 2024.1 we observed that `save_constraints -force` after
`create_debug_core` writes the probe connections but **omits the line
that connects the ILA's clock**. The result is an opt_design failure:

```
ERROR: [Chipscope 16-213] The debug port 'u_ila_0/clk' has 1
unconnected channels (bits). This will cause errors during
implementation.
```

Fix: open the autogenerated XDC and add the clock connection
explicitly, near the top of the debug section:

```tcl
set_property port_width 1 [get_debug_ports u_ila_0/clk]
connect_debug_port u_ila_0/clk [get_nets clk_IBUF_BUFG]
```

(`clk_IBUF_BUFG` is the post-IBUF/BUFG net name -- check the
synthesized netlist with `get_nets -of_objects [get_pins ...]` if
your top-level clock is named differently.)

## 4. Build

After `save_constraints` plus the manual clock fix, run a normal
`reset_run synth_1; reset_run impl_1` and then call `build.implement`
via the dispatcher. The ILA + dbg_hub are inserted automatically
during opt/place/route. On success:

- `bit` is generated as usual.
- `ltx` (`debug_nets.ltx`) now contains both VIO and ILA probe
  metadata.
- `program_device` reports `vios=N, ilas=1`.

## 5. Trigger setup

Trigger compare values use Vivado's own syntax, not Verilog:

```
<op><width>'<radix><value>
```

| Op | Meaning |
|---|---|
| `eq` | equal |
| `neq` | not equal |
| `lt` / `gt` / `lte` / `gte` | comparisons (signed/unsigned per radix) |

Value characters for 1-bit probes accept edges:

| Char | Meaning |
|---|---|
| `0`, `1`, `X` | level (X = don't-care) |
| `R` | rising edge |
| `F` | falling edge |
| `B` | both edges |

Examples:

| Want | Tcl value |
|---|---|
| Trigger when `mode_select == 1` (level) | `eq1'b1` |
| Trigger on rising edge of `mode_select` | `eq1'bR` |
| Trigger when `state_fast == 2'b11` | `eq2'b11` |
| Trigger when `fast_counter == 0xA5` | `eq8'hA5` |

### Tcl bracing matters

When passing trigger values through `exec_tcl.py`, the single-quote
in `eq1'bR` is interpreted by Tcl as a string delimiter unless you
enclose the whole token in `{...}`. **Always brace it:**

```bash
python exec_tcl.py "set p [get_hw_probes mode_select -of [get_hw_ilas hw_ila_1]]; set_property TRIGGER_COMPARE_VALUE {eq1'bR} \$p"
```

Without the braces you get
`ERROR: [Designutils 20-613] More than one apostrophe specified in string`.

### Other useful trigger properties

```tcl
# Where to put the trigger in the capture window (samples before trigger)
set_property CONTROL.TRIGGER_POSITION 1024 [get_hw_ilas hw_ila_1]
# Window size (must match what was synthesized via C_DATA_DEPTH)
set_property CONTROL.DATA_DEPTH 4096 [get_hw_ilas hw_ila_1]
# Capture mode: ALWAYS (every clock) or BASIC (only when capture cond true)
set_property CONTROL.CAPTURE_MODE ALWAYS [get_hw_ilas hw_ila_1]
# Trigger combine condition across multiple probes
set_property CONTROL.TRIGGER_CONDITION GLOBAL_AND [get_hw_ilas hw_ila_1]
```

`CONTROL.TRIGGER_MODE` is **read-only** in newer Vivado -- don't try
to set it; basic mode is the default.

### Trigger compare values persist across runs and AND together

This is a frequent trigger-setup footgun. **Once you `set_property
TRIGGER_COMPARE_VALUE` on a probe, that value stays on the probe for
every subsequent `run_hw_ila` until you explicitly change it.** With
the default `CONTROL.TRIGGER_CONDITION = GLOBAL_AND`, every probe's
compare value is AND'ed into the trigger condition. Stale conditions
on probes you forgot about silently block triggering.

Concrete example: you set `eq1'b1` on `mode_select` and capture. Later
you want to trigger on `eq1'bR` of `direction`. If you only set
`direction` and leave `mode_select` alone, the ILA waits for
`mode_select == 1 AND rising-edge of direction` -- not what you wanted.
And there is no error message; the ILA just never triggers.

**Use `ila.set_triggers` with `clear_others=true`** -- it resets
every other probe to don't-care first, then applies your conditions,
in one atomic call. This is exactly the pattern this API was added
to encode:

```bash
python vivado_op.py '{"op":"ila.set_triggers","params":{"values":{"direction":"rising"}}}'
# Every other probe on hw_ila_1 was reset to don't-care first, so
# stale conditions from earlier captures cannot contaminate this arm.
```

If you need to set values without clearing the others (rare), pass
`"clear_others": false` -- and accept the responsibility of tracking
what's still set on each probe. The default is `true` precisely
because forgetting to reset is the trap that made this API necessary.

## 6. Arm, drive, capture

The high-level flow is: pre-trigger state → arm → confirm armed →
drive the trigger → wait → export. Each step is one dispatcher call;
the `ila.arm` + `ila.wait_for_capture` pair encapsulates the polling
loop that we used to write by hand.

```bash
# 1. Make sure the design is in a "pre-trigger" state (host-side
#    sleep separates the write from the arm).
python vivado_op.py '{"op":"debug.write_vio_probe","params":{"probe":"mode_select_2","value":"0"}}'

# 2. Arm.
python vivado_op.py '{"op":"ila.arm","params":{}}'

# 3. Confirm armed.
python vivado_op.py '{"op":"ila.get_status","params":{}}'
# Expect status == "WAITING FOR TRIGGER"

# 4. Drive the trigger condition via VIO.
python vivado_op.py '{"op":"debug.write_vio_probe","params":{"probe":"mode_select_2","value":"1"}}'

# 5. Wait until full (or timeout).
python vivado_op.py '{"op":"ila.wait_for_capture","params":{"timeout":5.0}}'

# 6. Upload + export.
python vivado_op.py '{"op":"ila.export_csv","params":{"path":"capture.csv"}}'
```

If you absolutely need the raw Tcl form (e.g. for a status read that
the dispatcher doesn't expose), reach for the escape hatch:

```bash
python exec_tcl.py "get_property STATUS.CORE_STATUS [get_hw_ilas hw_ila_1]"
python exec_tcl.py "run_hw_ila [get_hw_ilas hw_ila_1]"
python exec_tcl.py "upload_hw_ila_data [get_hw_ilas hw_ila_1]"
python exec_tcl.py "write_hw_ila_data -force -csv_file {capture.csv} [current_hw_ila_data]"
```

### `wait_on_hw_ila` caveat

The Tcl command `wait_on_hw_ila -timeout 5` *should* return after
5 seconds, but in practice we've seen it block for 5 minutes if the
trigger never fires. The bridge's default 30-second exec_tcl timeout
then kills the connection mid-wait, leaving the Vivado side still
blocking. **Prefer polling** `STATUS.CORE_STATUS` from Python with a
short delay -- it gives you direct control over the timeout and
doesn't poison the Tcl Console.

`STATUS.CORE_STATUS` values:

| Value | Meaning |
|---|---|
| `IDLE` | Not armed |
| `WAITING FOR TRIGGER` | Armed, looking for trigger condition |
| `PRE-TRIGGER` | Trigger seen, capturing post-trigger samples |
| `FULL` | Capture complete, data ready to upload |

## 7. Reading the CSV

The exported CSV has a fixed shape:

```
Sample in Buffer,Sample in Window,TRIGGER,probe1,probe2,...
Radix - UNSIGNED,UNSIGNED,UNSIGNED,HEX,HEX,...
0,0,0,0,5,...
1,1,0,0,5,...
...
1024,1024,1,1,5,...    <-- TRIGGER row (TRIGGER column = 1)
...
```

- Row 0: column names.
- Row 1: per-column radix (HEX, BINARY, UNSIGNED, ...).
- Rows 2..N+1: sample data.
- The `TRIGGER` column is `1` exactly on the trigger sample, `0`
  elsewhere. With `TRIGGER_POSITION=1024` and `DATA_DEPTH=4096`,
  the trigger sample sits at index 1024 (1024 pre-trigger samples,
  3072 post).

Python parsing pattern:

```python
import csv
with open('capture.csv') as f:
    rows = list(csv.reader(f))
columns = rows[0]
radixes = rows[1]
samples = rows[2:]

# Find the trigger row
trig_idx = next(i for i, r in enumerate(samples) if r[2] == '1')

# Extract a column as integers, respecting radix
def col_int(name):
    j = columns.index(name)
    base = {'HEX': 16, 'BINARY': 2, 'UNSIGNED': 10, 'SIGNED': 10}[radixes[j]]
    return [int(r[j], base) for r in samples]

mode_select = col_int('mode_select')
counter     = col_int('fast_counter[7:0]')
```

## 8. VIO + ILA naming collision

### VIO outputs are auto-MARK_DEBUG'd

Before the renaming story: be aware that **any net connected to a VIO
`probe_out*` port gets `MARK_DEBUG=1` set on it as a side-effect of
the IP customization**. This is undocumented but consistent on Vivado
2024.1: open the synth run and `get_nets -filter {MARK_DEBUG==1}` will
include the VIO output nets even if you never wrote `(* mark_debug *)`
on them in HDL.

Why this matters: if you then add your own `mark_debug` to a different
signal set and rely on `create_debug_core`'s automatic probe pickup,
the resulting ILA will probe the **union** -- yours plus the VIO
outputs -- which is usually more probes (and more LUTs) than you
intended. To get only the probes you asked for, use explicit
`connect_debug_port` to a named net list rather than relying on
"all marked nets" pickup.

### Probe rename due to collision

When the same Verilog wire (e.g. `mode_select`) is connected to both
a VIO output **and** an ILA debug net, Vivado renames the VIO-side
runtime probe to avoid the collision.

Empirical observation on Vivado 2024.1 with our `led_blink` design:

| Source | Before adding ILA | After adding ILA |
|---|---|---|
| VIO output `mode_select` | `mode_select` | **`mode_select_2`** |
| VIO output `led_force`   | `led_force`   | **`led_force_2`** |
| VIO input `toggle`       | `toggle`      | **`toggle_1`** |
| VIO input `led_OBUF`     | `led_OBUF`    | **`led_internal_1`** |
| ILA probe `mode_select`  | (n/a)         | `mode_select` |

So adding an ILA **breaks any script that hardcoded VIO probe
names**. The only safe pattern is to call `debug.list_vio_probes`
**after** programming and use whatever names come back. SKILL guidance:
treat a fresh `program_device` like a fresh probe-name discovery.

### ILA runtime probe names follow the connected net, not `create_ila_core(name=...)`

The `name` field on each entry of `debug.create_ila_core`'s `probes`
list (`[{"name": ..., "nets": ...}, ...]`) becomes the *logical*
label, but at runtime (`ila.set_triggers`, `ila.parse_csv`,
`ila.list_ila_probes`) the names you see come from the underlying
nets in the synthesized design.

Two patterns to be aware of:

1. **Suffix appended on collision.** The net's basename gets a
   `_n` / `_1` etc. suffix when it is mark_debug-tagged or shares
   wires with another debug-visible signal. Example:
   `{name: "out_tvalid", nets: [u_fft/m_axis_data_tvalid]}` can come
   back at runtime as `out_tvalid_n` if the net has `mark_debug`
   applied.
2. **Full basename replacement.** When `nets` points at a register
   inside the design, the runtime probe name is the *register's*
   basename, not what you passed in `name`. Example:
   `{name: "counter_lo3", nets: [counter_reg[26], counter_reg[25],
   counter_reg[24]]}` can come back as `counter_reg`, completely
   overriding the user-supplied label.

The recovery is the same in both cases — call
`ila.list_ila_probes` after `program_device` and use the names
that come back; never hard-code from the `create_ila_core` argument
list. The same applies to VIO probes (using_vio.md §2).

## 9. Cheat sheet

| Task | Tcl |
|---|---|
| List ILA cores | `get_hw_ilas` |
| List ILA probes (and find runtime names) | `get_hw_probes -of [get_hw_ilas hw_ila_1]` |
| Set rising-edge trigger | `set_property TRIGGER_COMPARE_VALUE {eq1'bR} $p` |
| Set level trigger | `set_property TRIGGER_COMPARE_VALUE {eq1'b1} $p` |
| Trigger position | `set_property CONTROL.TRIGGER_POSITION 1024 [get_hw_ilas ...]` |
| Arm | `run_hw_ila [get_hw_ilas ...]` |
| Status | `get_property STATUS.CORE_STATUS [get_hw_ilas ...]` |
| Upload | `upload_hw_ila_data [get_hw_ilas ...]` |
| Export CSV | `write_hw_ila_data -force -csv_file {x.csv} [current_hw_ila_data]` |
| Export VCD | `write_hw_ila_data -force -vcd_file {x.vcd} [current_hw_ila_data]` |

## 9.5 Auto-populating the ILA Waveform pane

Captured data exports cleanly via `write_hw_ila_data -csv_file`,
which is what the AI loop consumes. But if the user *also* wants
to look at the waveform in the Vivado IDE without right-clicking
"Add Probes", you can install the Waveform pane layout (which probes,
in what order, with which radix, Digital or Analog) by editing the
project's `hw.xml` and the matching `.wcfg` outside Vivado.

`scripts/setup_dashboard.py` does this from a JSON layout file. The
mechanism, the trade-offs, and the format are covered in the dedicated
guide: [using_dashboard_hack.md](using_dashboard_hack.md). The short
version: drive `close_hw_manager` -> rewrite both files -> re-open
the Hardware Manager and refresh the device.

## 10. Things this document doesn't cover yet

- **Advanced trigger state machines** (`CONTROL.TSM_FILE`, `.tsm`
  language). Useful for "trigger after N occurrences" or
  "A then B" sequences. The basic flow above covers the common case.
- **Capture-mode storage qualification** (`CAPTURE_COMPARE_VALUE` on
  probes, `CONTROL.CAPTURE_MODE BASIC`). Lets you keep only samples
  matching a condition, stretching the effective window.
- **TRIG_IN / TRIG_OUT cross-ILA chaining**.

These all reuse the same property/Tcl pattern and can be added
incrementally to `operations.ila` without changing the bridge itself.

## 10b. Pitfalls discovered during real bring-up

Several traps that cost real time in practice; all of them are now
either avoided by `operations.ila` or documented here so you don't
re-discover them.

### `mark_debug` alone does NOT insert an ILA (Vivado 2024.1)

**This is the single biggest footgun in this flow.** It is a
recurring case: read this doc, decide to use `mark_debug` alone,
build and program the bitstream, and get `ila_count=0` back from
`program_device`. The bitstream is clean; the ILA simply isn't there.

**The rule:** to get an ILA into your bitstream you need TWO things:

1. `mark_debug` (Verilog attribute or XDC) -- this only **preserves
   the net through synth optimization**. It marks candidates; it does
   not materialise a core.
2. **`create_debug_core` somewhere** -- in a debug XDC, in the main
   XDC, or via Tcl on the open synthesized run. This is what actually
   instantiates the ILA hardware in the design.

If you skip step 2, the netlist will have `MARK_DEBUG==1` nets
(visible via `get_nets -filter {MARK_DEBUG==1}` after synth) but no
ILA core. Implementation will run cleanly, the bitstream will program
fine, and `program_device` will report `ilas=0`. There is no error
message anywhere -- the run just silently produces a bitstream with
no debug core in it.

**Recommended pattern: `debug.create_ila_core(...)`**

The bridge wraps the whole sequence -- core creation, per-probe ports,
dbg_hub clock fix, dedicated XDC routing -- in one call:

```bash
python vivado_op.py '{"op":"build.open_synth","params":{}}'

python vivado_op.py '{"op":"debug.create_ila_core","params":{
  "name": "u_ila_0",
  "clock_net": "clk_IBUF",
  "probes": [
    {"name": "count",     "nets": ["count[0]","count[1]","count[2]","count[3]","count[4]","count[5]","count[6]","count[7]","count[8]","count[9]","count[10]","count[11]","count[12]","count[13]","count[14]","count[15]"]},
    {"name": "en",        "nets": "en"},
    {"name": "direction", "nets": "direction"}
  ],
  "depth": 4096,
  "dbg_hub_clock_freq_hz": 125000000
}}'

python vivado_op.py '{"op":"build.close_design","params":{}}'
python vivado_op.py '{"op":"build.implement","params":{}}'
```

Equivalent hand-rolled Tcl (in a dedicated `debug_<name>.xdc`):

```tcl
create_debug_core u_ila_0 ila
set_property C_DATA_DEPTH 4096       [get_debug_cores u_ila_0]
set_property C_TRIGIN_EN  false      [get_debug_cores u_ila_0]
set_property C_TRIGOUT_EN false      [get_debug_cores u_ila_0]

set_property port_width 1 [get_debug_ports u_ila_0/clk]
connect_debug_port u_ila_0/clk [get_nets clk_IBUF]

set_property port_width 1 [get_debug_ports u_ila_0/probe0]
connect_debug_port u_ila_0/probe0 [get_nets mode_select]
# ... more probes ...

# dbg_hub clock -- override the 300 MHz default to your real clock
set_property C_CLK_INPUT_FREQ_HZ 125000000 [get_debug_cores dbg_hub]
set_property C_USER_SCAN_CHAIN 1           [get_debug_cores dbg_hub]
connect_debug_port dbg_hub/clk             [get_nets clk_IBUF]
```

Why the dedicated XDC: with `target_constrs_file` left at the default
(typically the user's `pynq_z1.xdc` or similar), `save_constraints`
would append the ~30 lines above into the user-authored file -- a
recurring pollution case. `create_ila_core` routes the save into
`<project>.srcs/constrs_1/imports/debug_<name>.xdc` instead, then
restores `target_constrs_file` to its previous value.
The dedicated XDC also makes `delete_ila_core` a clean
file-removal operation rather than a stripping-of-text-from-a-mixed-
file operation.

### Don't call `implement_debug_core`

Once you have `create_debug_core` in an XDC (per the rule above),
Vivado materialises the ILA automatically during `opt_design`. Calling
`implement_debug_core` explicitly does the equivalent of "materialise
the core into the in-memory synth_1 design", which can take tens of
seconds to minutes -- well past the bridge's default exec_tcl timeout,
after which the call looks like it failed even though Vivado is still
working.

The clean recipe is:

```tcl
# After synth_1 finishes:
save_constraints -force        ;# writes debug constraints into a constrs_1 XDC
close_design                   ;# drop the in-memory synth view
launch_runs impl_1 -to_step write_bitstream -jobs 4
```

No `implement_debug_core`.

### Some `CONTROL.*` properties are read-only

`CONTROL.CAPTURE_MODE`, `CONTROL.TRIGGER_CONDITION`, and `TRIGGER_MODE`
are listed as configurable but are read-only on cores synthesized
without storage qualification or with a single trigger probe.
`set_property` returns `[Labtools 27-158] cannot be set to invalid
value` on those cores. The minimal write set that always works is
`CONTROL.DATA_DEPTH` + `CONTROL.TRIGGER_POSITION`. `operations.ila`
exposes only those two. If you genuinely need the others, send the
raw `set_property` yourself and let Vivado reject it loudly.

### Self-driving stimulus when you can't press a button

For unattended automation flows, the design needs to drive its own
"start" signal -- you can't have ILA arm waiting for a button push
that never comes. The bitstream-program-then-arm sequence takes ~10s
of seconds; a "one-shot start 1 second after reset" generator fires
*before* arm completes and never re-fires. Use a periodic generator
instead:

```verilog
// 1 Hz auto-start at 125 MHz sysclk
reg [27:0] auto_cnt = 0;
reg        auto_pulse;
always @(posedge clk) begin
    auto_pulse <= 1'b0;
    if (rst) auto_cnt <= 0;
    else if (auto_cnt == 28'd125_000_000) begin
        auto_pulse <= 1'b1;
        auto_cnt   <= 0;
    end else
        auto_cnt <= auto_cnt + 1;
end
wire start = btn_start | auto_pulse;
```

Plus relax the FSM's done-guard so the next `start` is accepted even
while `done` is latched from the previous run (`if (start && !busy)`
rather than `if (start && !busy && !done)`). With program-then-arm
taking <10s and the period set to 1s, every arm is guaranteed to land
on the next rising auto-start.

### CSV column names carry the bit-slice suffix

Vivado writes ILA CSV headers like `dbg_out_x[6:0]`, not `dbg_out_x`.
A naive `header.index("dbg_out_x")` raises ValueError. Use a partial
match: the `ila.find_column` dispatcher op does this for you, and
returns the column index of the first header whose base name (before
the `[`) equals the target. The same goes for radix annotation rows
(`Radix - UNSIGNED,UNSIGNED,...`) -- `ila.parse_csv` already skips
them.
