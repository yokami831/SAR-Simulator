# Git management for Vivado projects

A practical guide for putting a Vivado project under version control
without dragging gigabytes of generated artifacts into the repo.

## Why this needs a guide

A typical Project Mode workspace looks unassuming -- a `.xpr` file
plus a few subdirectories -- but the subdirectories grow without
warning. Measured on a single Vivado 2024.1 PYNQ-Z1 design that has
done a few synth/impl + sim runs:

```
project_1/
├── project_1.xpr               (15 KB)
├── srcs/                       (25 KB)        own HDL / XDC
├── project_1.srcs/             (232 KB)       project-internal sources + .dcp
├── project_1.hw/               (413 KB)       Hardware Manager state
├── project_1.gen/              (3.2 MB)       generated IP outputs
├── project_1.runs/             (17 MB)        synth/impl run logs
├── project_1.cache/            (23 MB)        IP cache
└── project_1.sim/              (6.4 GB)       XSim runtime data
                                ───────
                                ≈ 6.4 GB
```

`git add .` on that tree pushes a `.gitignore` problem into the
"my disk is full" category. AMD's official guidance (XAPP1165,
UG1198) is roughly: don't commit the project, commit a recipe that
re-creates it. This document adapts that guidance for the way this
skill is actually used -- Project Mode, AI-driven, single FPGA
designer working on small to mid-size designs.

For perspective: applying the Level A `.gitignore` from this guide
to that same 6.4 GB project shrinks the tracked set to **229 KB**
across **7 files** (the HDL, the XDC, one VIO `.xci`, the `.xpr`).
That's roughly a 30,000× reduction, achieved with no scripting --
just a `.gitignore`. The rest of this guide explains how to get
that, and what the next levels of effort buy you on top.

### Vivado 2022.1 changed the layout in our favor

Before Vivado 2022.1 the project tree mixed your sources and Vivado's
generated wrappers under the same `*.srcs/` directory, which made
".gitignore the generated parts" awkward. From 2022.1 onward the
generated outputs live in a parallel `*.gen/` directory, leaving
`*.srcs/` for things you actually authored (HDL, XDC, the IP `.xci`).
The `.gitignore` below assumes that split. If you're on an older
Vivado, you may need to be more selective about what to keep inside
`*.srcs/`.

### A note on `.xpr`

Some older write-ups call `.xpr` "binary" and warn against tracking
it. On Vivado 2022.1+ it's plain XML (15 KB on the project we
measured -- you can `cat` it, `git diff` reads cleanly, merge
conflicts are tractable). Tracking it is a real option; the
trade-off is described in the next section.

## Two principles

1. **Commit the recipe and the inputs, not the outputs.** The
   inputs are HDL files, XDC files, IP customization (`*.xci`), and
   a TCL script that re-creates the project structure. Everything
   else can be regenerated by Vivado from those.

2. **Layer your effort to taste.** You don't have to do all of it
   at once. The big win is just `.gitignore` (level A below); the
   recipe step (level B) is what makes the repo portable across
   machines. Pick the level that matches how much you care about
   reproducibility.

## Level A: just `.gitignore` (the 90%)

Drop a `.gitignore` next to the `.xpr` (or at the repo root if the
project sits at the repo root). This alone keeps the repo from
growing without bound:

```gitignore
# ---------- Vivado generated state ----------
# These directories are regenerated by Vivado from the recipe + sources.
# Sizes observed on a small Zynq-7 design: 6+ GB combined.
*.cache/
*.runs/
*.hw/
*.sim/
*.gen/
*.ip_user_files/
.Xil/

# ---------- Vivado scratch / log files ----------
*.jou
*.log
*.str
*.pb
*.wdb
*.backup.*
*.dcp

# ---------- Inside project_1.srcs/, ignore generated IP outputs ----------
# Keep the .xci (IP customization) but skip the generated wrappers and
# behavioural simulation models that come with it.
**/ip/*/*
!**/ip/*/*.xci

# ---------- HW Manager screenshot output (this skill) ----------
screenshots/
```

Notes on the choices:

- `*.dcp` (design checkpoint) is regenerated by `synth_design` /
  `place_design` etc. It's tens of MB and binary. Some teams check
  in a `_post_synth.dcp` for archival; that's a deliberate decision,
  not a default.
- `*.ip_user_files/` is fully derivable from the `.xci` files plus
  Vivado's IP catalog. Don't track it.
- The `**/ip/*/` rule keeps `vio_0/vio_0.xci` but drops the
  `vio_0/synth/`, `vio_0/sim/`, `vio_0/hdl/` directories Vivado
  fills in when you generate IP. Those are regenerated on
  `generate_target all`.
- `screenshots/` matches the bridge's own `capture_screenshot.py`
  rolling output. If you don't use that, drop the line.

After applying this, `git status` shows you HDL, XDC, the `.xci`
files, and the `.xpr` itself. That's already a usable repo for one
person on one machine -- you'll see meaningful diffs of what *you*
edited, with no noise from synth runs.

This is the approach the recent Vivado-on-git write-ups land on
(VHDL Whiz, RTL Audio Lab, Adiuvo's *MicroZed Chronicles*,
fpgalover.com, the github/gitignore community Xilinx template), and
on Vivado 2022.1+ it works cleanly thanks to the `.srcs/.gen`
split. What it gives you, and what to be aware of:

- **Lower maintenance.** You commit what Vivado already saved when
  you hit Ctrl+S. There is no separate "regenerate the recipe"
  step to remember.
- **Diff-readable structure changes.** Adding a source, switching
  the top, changing a strategy all show up as XML diffs in
  `.xpr`. Reviewable, mergeable.
- **Version-locked.** A `.xpr` saved by Vivado 2024.1 may need a
  one-time upgrade dialog when opened with 2024.2, and re-opening
  it with a much older Vivado is not safe. If your project lives
  in one Vivado version anyway (the common case for a personal
  board project), this isn't a problem.
- **Path-locked-ish.** The `.xpr` references `.xci` and HDL paths
  relatively to itself, so moving the whole repo around is fine.
  But if you've used "Copy sources into project" and your
  collaborators clone to a different absolute path, the file
  references inside the project survive only because they live
  under `*.srcs/` and get tracked along with `.xpr`. If you didn't
  copy them in, you need Level B.

If those trade-offs don't bite, Level A is the realistic answer.
Level B exists for the cases where they do.

## Level B: sources outside the project + recipe TCL

Level A keeps the repo small but doesn't make the project portable.
The next problem is "my coworker clones the repo, opens the `.xpr`,
and Vivado complains about absolute paths or missing files." Level B
fixes that.

You probably want Level B when one of these starts to bite:

- **Multiple developers on different OSes / different install paths.**
  A `.xpr` saved with absolute paths won't work for them.
- **You change Vivado versions.** Re-creating from a recipe is
  tidier than fighting upgrade dialogs on a tracked `.xpr`.
- **A CI job needs to build from a clean clone.** Recipe-driven
  builds are the natural fit.
- **You want to publish the repo (OSS, sharing on GitHub, etc.).**
  Strangers will not have your directory layout.

If none of those apply, Level A is fine and Level B is overhead.

### Step 1: keep your sources outside the Vivado project directory

When you create a new project, **uncheck "Copy sources into project"**
in the Add Sources dialog. Place your HDL / XDC / IP `.xci` outside
the project directory:

```
my_repo/
├── src/
│   ├── hdl/                    your *.v / *.sv / *.vhd
│   ├── constraints/            your *.xdc
│   └── ip/                     your *.xci  (one dir per IP)
├── sim/                        testbenches
├── script/
│   └── create_project.tcl      <-- written by export_project.py
├── .gitignore
└── project/                    <-- the Vivado project dir
    ├── project_1.xpr           (gitignored, regenerable)
    └── project_1.{cache,runs,hw,sim,gen}/   (gitignored)
```

The point is that *every* file you commit lives in a directory
**outside** `project/`. The `project/` tree is fully expendable.

If your existing project has sources copied inside the project
directory (the default in fresh Vivado projects), the migration is
manual and a little tedious: move the files out, re-add them with
"Copy sources into project" unchecked, regenerate the recipe. The
skill won't do this for you because it isn't a project-editing tool.

### Step 2: write a recipe with `export_project.py`

```bash
mkdir -p script
python <bridge>/scripts/export_project.py script
```

This produces `script/create_project.tcl` (and `script/bd/<name>.tcl`
for any block designs in the project). What's in `create_project.tcl`:

- Project settings (target part, board, default runs, strategies)
- A list of source files referenced via portable
  `[file normalize "$origin_dir/..."]` paths
- IP creation via `read_ip` against the same `.xci` files

The recipe references your `src/` files relative to itself, so a
clone on another machine works the same way.

### Step 3: re-create the project on a fresh clone

```bash
git clone <repo>
cd <repo>
vivado -mode batch -source script/create_project.tcl
```

Vivado regenerates `project/project_1.xpr` and the empty
`project_1.{cache,runs,hw,sim,gen}/` directories. Open the project,
launch synth/impl, and you're back where the previous user left the
repo.

### When to re-export

Run `export_project.py` again whenever you've made a *structural*
change to the project that you want preserved across clones:

- added or removed a source file
- added or removed an IP
- changed the target part or board
- changed a synthesis or implementation strategy
- changed the top module

You don't need to re-export for plain HDL edits (those are tracked
through `src/`) or for run results (those are regenerated).

## Level C: every commit reproducible (CI-ready)

If you want a CI job to take a fresh clone and produce the bitstream,
on top of Level B you need:

- **Re-exporting the recipe before every commit** that has touched
  project structure. A simple shell wrapper (`commit_helper.sh` in
  the repo) that runs `export_project.py` and then `git add script/`
  is enough. Don't put `export_project.py` in a git pre-commit hook
  unless you can absorb a 30-60 second Vivado startup on every
  commit; the hook also won't work when the bridge isn't running.
- **A CI script** that runs `vivado -mode batch -source
  script/create_project.tcl` then `vivado -mode batch -source
  script/build.tcl` (the latter is your own thin wrapper that
  launches synth_1 / impl_1 / write_bitstream).
- **Tracking absolute-path leaks**. `write_project_tcl
  -no_copy_sources` produces portable executable Tcl on Vivado
  2024.1, but the header comments include host-side absolute paths
  for human reference. If your repo is public, scrub them, or
  accept that someone's username appears in the comment block.

Level C is rarely necessary for a one-person FPGA project. It pays
off when you're sharing the repo with collaborators, running a CI
build, or trying to reproduce a release months later.

## What to commit / ignore -- quick reference

| Path | Track? | Rationale |
|---|---|---|
| `*.xpr` | optional (Level A: yes; Level B: no) | tiny XML, but Level B regenerates it from TCL |
| `*.tcl` you wrote | yes | this is the recipe |
| `script/create_project.tcl` | yes | the auto-generated recipe |
| `script/bd/<name>.tcl` | yes | block design recipes |
| HDL (`*.v`, `*.sv`, `*.vhd`) | yes | the design |
| XDC | yes | constraints |
| `*.xci` | yes | IP customization, the input to `generate_target` |
| `*.bd` (raw XML) | discouraged | track the `write_bd_tcl` output instead |
| `*.dcp` | no by default | regenerable from synth/impl runs |
| `*.bit`, `*.ltx` | no | regenerable from impl + write_bitstream |
| `*.cache/`, `*.runs/`, `*.hw/`, `*.sim/`, `*.gen/` | no | generated state |
| `*.ip_user_files/` | no | derivable from `.xci` |
| `.Xil/` | no | Vivado scratch |
| `*.jou`, `*.log`, `*.str`, `*.wdb` | no | per-process logs |

## Anti-patterns

- `git add .` in a project directory before reading this guide.
  You will commit gigabytes.
- Committing `*.dcp` "just in case". It's binary, large, and
  regenerable. If you actually need an archival snapshot, tag the
  commit and let CI rebuild from source.
- Putting the recipe TCL inside `project/` so Vivado overwrites it
  on the next save. Keep it outside (`script/`).
- Editing `create_project.tcl` by hand. It's regenerated by
  `export_project.py`; hand edits get lost on the next export.
- Treating `*.bd` as a source file. The XML is unreadable in diffs
  and merges are painful. Use `write_bd_tcl` (already in
  `export_project.py` for any BD it finds).

## Pitfalls AIs hit

- **The header in `create_project.tcl` lists absolute source paths**
  in a comment block at the top. The executable Tcl below is
  portable, but the header still says e.g.
  `# "D:/Users/alice/.../my_proj.srcs/.../top.v"`. Don't worry
  about the header for private repos; for public repos, either
  scrub the header, regenerate from a project where sources sit
  outside the project dir (Level B), or accept the username leak.
- **`-no_copy_sources` is required.** Without that flag,
  `write_project_tcl` writes a recipe that copies sources into a
  fresh project directory at re-create time, which defeats Level B.
  `export_project.py` always passes `-no_copy_sources`; if you call
  `write_project_tcl` directly, remember to pass it too.
- **Block design XML drifts.** If you commit `*.bd` directly,
  Vivado may rewrite it on every open even when you change nothing,
  producing spurious diffs. The TCL form (`write_bd_tcl`) is
  text-friendly and stable.
- **`generate_target` is not run automatically by the recipe.**
  After re-creating a project, you may still need to call
  `generate_target all [get_files *.xci]` (or just open the IP and
  let Vivado prompt) before synthesis works. The recipe creates
  the IP entry but doesn't trigger generation.

## When this approach isn't enough

If you outgrow the script-and-gitignore approach -- multiple
designers, multiple boards, automated releases -- there are
established frameworks that build on the same XAPP1165 philosophy
with more structure:

- [Hog](https://hog.readthedocs.io/) (CERN) -- TCL/shell submodule
  with CI/CD integration, version-tagging, multi-project release
  flow. Mature but heavy.
- [barbedo/vivado-git](https://github.com/barbedo/vivado-git) --
  lightweight wrapper around `write_project_tcl` with
  absolute-path scrubbing, callable directly from the Vivado Tcl
  Console.

Both go beyond what this skill needs to provide. They're listed
here so you know they exist; this guide stays focused on the
single-developer / single-project case.
